# driftd — build devlog

AI-powered anti-tourist city exploration. Walking routes generated by Claude, narrated by an AI companion. Free for tourists, cities pay.

**Stack:** Next.js 14 (App Router) · TypeScript · Tailwind CSS · Vercel  
**Cities:** Warsaw · Berlin · Prague  
**Repo:** github.com/nexooka/driftd

## what got built

### route generation (`/api/generate-route`)

The core API. Takes city, vibes, time, starting point → returns a full walking route as JSON.

**Claude model routing:**
- Routes ≤90 min → `claude-sonnet-4-6`
- Routes >90 min → `claude-haiku-4-5-20251001` (faster, cheaper for longer output)

**Server-side pipeline (in order):**
1. Geocode starting address via Nominatim
2. Lock stop #1 to user's exact coords (server-enforced, Claude can't override)
3. Geocode all other stops via Nominatim (multiple query variants to handle names like "Grébovka (Havlíčkovy sady)")
4. Nearest-neighbor reorder (greedy TSP) — always uses real coords, not Claude's hallucinated ones
5. 2-opt improvement pass — eliminates crossing paths and zigzag patterns
6. Remove outlier stops (walks >22 min, keeps ≥ minutes/10 stops)
7. Enrich with haversine walk times (pure math, always consistent — no OSRM server-side)
8. Scale stop times to fill time budget

**Key rules baked into the system prompt:**
- River rule: never place stops on both sides of a major river
- Walk limit: 600m straight-line max between adjacent stops
- Stop count minimums: 100-min route = minimum 11 stops
- Walk notes: mandatory for any walk ≥6 minutes
- Knowledge base only: no invented business names, ever
- 70% niche stops minimum

**Walking math:**
```
meters = haversine(a, b) × 1.35   // road correction factor
minutes = max(1, round(meters / 83))  // 83 m/min = 5 km/h
```

### map (`/components/RouteMap.tsx`)

Leaflet.js with CartoDB Dark Matter tiles. Single polyline per segment (removed glow layer — it caused zoom-level glitches). OSRM public API used client-side only for actual road-following geometry.

- Markers: amber circles with stop numbers, larger for first/last
- Route: amber polyline, `smoothFactor: 3` for cleaner appearance
- Dashed line fallback when OSRM returns no geometry
- User can tap markers to "walk" the route step by step (segments dim/highlight)

### demo page (`/demo`)

Full route generation UI. Three views: form → loading → result.

**Form:**
- City picker (Warsaw / Berlin / Prague) with SVG city silhouettes
- Vibe tags (artsy, historic, foodie, nightlife, sketchy-but-cool, chill, romantic, photogenic, quirky) — max 3
- Time slider (10–240 min)
- Starting point + optional destination + optional notes
- Rate limit: 5 generations per minute (client-side, localStorage)

**Loading:**
- "d" logo with breathing amber glow
- City skyline SVG with flickering windows
- Animated stop dots with expanding rings
- Walker dot on animateMotion path between dots
- Rotating loading messages

**Result:**
- Route header: city, vibes, total time, stop count, total walking distance
- Intro paragraph (all prose rendered lowercase)
- Leaflet map with route
- Stop cards: name, neighborhood, tagline, description, why_this_spot, address, time at stop
- Walk connectors between cards: walk time + distance, expandable walk note on hover
- "See photo" button → Google Places API → Street View fallback
- "More info" button → `/api/stop-info`
- Actions: drift again · open in google maps · share · save this drift (email)

**Share button:** Web Share API on mobile, clipboard copy on desktop. Formats route as readable text (stop names + walk times + driftd.world attribution).

## bugs fixed

| Bug | Root cause | Fix |
|---|---|---|
| ESC not closing photo lightbox | handler only wired to `showModal`, not `photoModal` | added `setPhotoModal(null)` to the keydown handler |
| 7km shown as 10 min walking | OSRM server-side returning inconsistent duration/distance from Vercel IPs | removed all server-side OSRM, pure haversine everywhere |
| Wrong geocode for stops with parenthetical names | Nominatim failing on "Grébovka (Havlíčkovy sady)" | try multiple query variants: full name, name before paren, name inside paren, address |
| Route ordering wrong (far stop #2, nearby stop #3) | `reorderStops` ran before `geocodeStops` — sorted on Claude's hallucinated coords | moved geocoding before reordering |
| Map line visual glitches at different zoom levels | Two overlapping SVG polylines (glow + main) | removed glow layer entirely, single polyline per segment |
| Routes zigzagging | Nearest-neighbor doesn't remove crossing paths | added 2-opt improvement pass after nearest-neighbor |
| 100-min route returning 5 stops | `enrichAndAdjust` floor was `minutes/20` = 5, prompt target was too low | raised floors, strengthened prompt with explicit minimums |
| "Join waitlist" button broken on `/demo` | `#waitlist` anchor only exists on homepage; Next.js client routing drops hash | `window.location.href = '/#waitlist'` for non-home pages |

## design decisions

**Why haversine not OSRM server-side:**
OSRM from Vercel IPs was returning correct distances but wrong durations — same formula, inconsistent results. Haversine always produces mathematically consistent meters and minutes from the same input. Accuracy traded for reliability.

**Why single polyline per segment:**
Two overlapping SVG elements (glow layer + main layer) rendered differently depending on zoom level — sometimes only the glow was visible, sometimes only the main line. Single element, no layering, no glitches.

**Why geocode before reorder:**
Claude's coordinates are unreliable (it hallucinates locations). If you sort by nearest-neighbor using hallucinated coords, the route order is wrong even after geocoding fixes the individual points. Geocode first, sort second.

**Why lowercase prose:**
Fits the brand voice — casual, texting-a-friend tone. Applied at render time so the API output is unchanged. Names, addresses, neighborhoods left as-is.
